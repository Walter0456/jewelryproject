Executive Summary`nJewelAdmin Pro is a full-stack POS and inventory management system for Rodriguez Jewelry, built with React (frontend), Node.js/Express (backend), and PostgreSQL (database). The system manages sales, products, staff access, audit logs, and financial reports.`n`nThis audit was performed on the complete packed codebase (Repomix export). Analysis covers the backend API server, database schema, frontend components, utilities, and deployment scripts. A prior security audit was embedded in the repository (February 12, 2026) — the current code shows that many of those critical issues have since been remediated. This report focuses on what remains unresolved and newly discovered issues.`n`n2. Remediation Progress (Previously Fixed)`nThe following critical vulnerabilities from the prior audit have been successfully addressed:`n`nSQL Injection — All queries now use parameterized statements ($1, $2 placeholders). ✅`nHardcoded DB Credentials — Moved to .env / .env.local environment variables. ✅`nMD5 Password Hashing — Replaced with PBKDF2 (100,000 iterations, SHA-256). ✅`nJWT Authentication — Implemented with authenticate() middleware and requireAdmin() guard. ✅`nRate Limiting — loginLimiter (10/15min), pinLimiter (10/5min), qrLimiter added. ✅`nCommand Injection — backup uses execFile() (not exec), preventing shell injection. ✅`nFile Upload Restriction — MIME type + extension validation; 5 MB size limit. ✅`nMissing JWT_SECRET check — Server exits at startup if JWT_SECRET is not set. ✅`n`n`n3. Detailed Findings`n`n#1 — Unauthenticated Audit Log Write  [CRITICAL]`nAffected File(s)`nbackend/server.js — POST /api/logs (~line 1979)`nProblem`nThe POST /api/logs endpoint has no authenticate middleware. Any device on the network can inject arbitrary entries into the activity audit trail by sending a simple HTTP request. Because the system relies on these logs for staff accountability and inventory auditing, forged entries completely undermine trust in the audit trail.`nFix`nAdd the authenticate middleware to this route. Additionally consider adding requireAdmin since only trusted code paths should insert activity log entries — or better, move all log insertion server-side and never expose a public log-write endpoint.`nCode`n// ❌ CURRENT (no auth):`n`napp.post(&apos;/api/logs&apos;, async (req, res) => { ... });`n`n`n`n// ✅ FIX:`n`napp.post(&apos;/api/logs&apos;, authenticate, async (req, res) => { ... });`n`n#2 — Weak Seeded Admin Password Hash + Hardcoded Salt  [CRITICAL]`nAffected File(s)`nschema.sql (line 767–769) — also backend/schema.sql (line 767–769)`nProblem`nThe default &apos;admin&apos; user is seeded with hash &apos;8c6976e5b5410415bde908bd4dee15dfb167a9c873fc4bb8a81f6f2ab448a918&apos; — this is simply SHA-256(&apos;admin&apos;), not PBKDF2. The salt is also hardcoded: &apos;c3ab8ff13720e8ad9047dd39466b3c89&apos;. If an attacker obtains a database backup, they can immediately identify the hash is SHA-256 and crack &apos;admin&apos; in milliseconds using any rainbow table. The rest of the application uses PBKDF2, but the bootstrap account bypasses it.`nFix`nRemove the pre-hashed credential seed. Instead, on first startup the server should detect no admin exists, generate a PBKDF2 hash of a strong random password (printed once to stdout), and insert that. Force an immediate password change on first login. Never commit known-crackable hashes to source control.`nCode`n// ❌ CURRENT — SHA-256 of literal string &apos;admin&apos;:`n`nINSERT INTO users (username, password_hash, salt, role, status)`n`nVALUES (&apos;admin&apos;, &apos;8c6976e...918&apos;, &apos;c3ab8ff...18&apos;, &apos;admin&apos;, &apos;active&apos;)`n`n`n`n// ✅ FIX — generate at startup (pseudocode):`n`nconst tmpPass = crypto.randomBytes(16).toString(&apos;hex&apos;);`n`nconst { hash, salt } = await hashPassword(tmpPass);`n`nconsole.log(&apos;FIRST RUN: Admin password =&apos;, tmpPass);`n`n// Insert hash+salt into DB, set must_change_password = true`n`n#4 — Legacy PIN Plaintext Fallback (Timing Attack)  [CRITICAL]`nAffected File(s)`nbackend/server.js — verifyPin() function (~line 1025–1037)`nProblem`nverifyPin() has a fallback: if the stored PIN does not start with the &apos;pbkdf2$&apos; prefix it returns stored === pin — a plain string equality check. This means (a) any legacy unupgraded PINs are compared in plaintext making them trivially readable if the DB is compromised, and (b) the comparison is not timing-safe, allowing a remote timing attack to enumerate valid PINs character by character.`nFix`nRemove the plaintext fallback entirely. All PINs must be hashed with PBKDF2 before storage. Use crypto.timingSafeEqual() for ALL comparisons. Run a one-time migration to force re-entry of any PINs stored in legacy format.`nCode`n// ❌ CURRENT — plaintext fallback:`n`nif (!stored.startsWith(PIN_HASH_PREFIX)) {`n`n  return stored === pin; // timing-unsafe, plaintext!`n`n}`n`n`n`n// ✅ FIX — reject legacy PINs:`n`nif (!stored.startsWith(PIN_HASH_PREFIX)) {`n`n  return false; // Force re-set`n`n}`n`n#5 — Unauthenticated GET /api/settings  [CRITICAL]`nAffected File(s)`nbackend/server.js — GET /api/settings (~line 1920)`nProblem`nThe settings endpoint returns branch configuration (branch name, tax rate, low stock threshold, session timeout, etc.) without requiring a valid JWT token. While admin_pin is masked, the response reveals operational parameters useful to an attacker scoping the system. It also exposes branchName and business metadata.`nFix`nAdd the authenticate middleware to the GET /api/settings route. Settings are only needed by logged-in users.`nCode`n// ❌ CURRENT:`n`napp.get(&apos;/api/settings&apos;, async (req, res) => { ... });`n`n`n`n// ✅ FIX:`n`napp.get(&apos;/api/settings&apos;, authenticate, async (req, res) => { ... });`n`n#6 — Wildcard CORS Configuration  [CRITICAL]`nAffected File(s)`nbackend/server.js (~line 985)`nProblem`napp.use(cors()) with no options sets Access-Control-Allow-Origin: * — any website can make cross-origin requests to the API. On a local network this means a rogue page opened on any browser connected to the same WiFi (e.g., customer using in-store WiFi) can silently call the API if they can obtain a stolen JWT from localStorage. Combined with the client-side session storage, this creates a meaningful attack surface.`nFix`nRestrict CORS to the specific origin(s) that should access the API. For a LAN-only deployment, whitelist the local frontend origin explicitly.`nCode`n// ❌ CURRENT:`n`napp.use(cors());`n`n`n`n// ✅ FIX:`n`napp.use(cors({ origin: &apos;http://localhost:5173&apos;, credentials: true }));`n`n#7 — Developer PII Committed to Repository  [CRITICAL]`nAffected File(s)`nscripts/snapshot-inventory-task.xml`nProblem`nThe Windows Task Scheduler XML file hard-codes a developer&apos;s personal path: C:\Users\Walter\Desktop\jeweladmin-pro\... and their name &apos;Walter&apos; as the task Author. This is committed to version control, exposing the developer&apos;s real name, machine username, and file system structure. If this repository is ever shared or leaked, this constitutes a personal data disclosure.`nFix`nReplace the hardcoded paths with relative paths or environment variables. On V8 (Node.js), its internal state can potentially be predicted after observing several outputs, allowing an attacker to pre-calculate future codes and self-register unauthorized accounts.`nFix`nUse crypto.randomBytes() to generate registration codes.`nCode`n// ❌ CURRENT:`n`nconst code = Math.random().toString(36).substring(2, 8).toUpperCase();`n`n`n`n// ✅ FIX:`n`nconst code = crypto.randomBytes(4).toString(&apos;hex&apos;).toUpperCase(); // 8 hex chars`n`n`n#9 — XSS in Print Service  [HIGH]`nAffected File(s)`nutils/printService.ts`nProblem`nThe printFullReport() function opens a popup window and calls document.write() with unsanitized data including title, metadata.branch, headers[], and rows[]. If any product name, branch name, or report title contains HTML/script tags, it executes in the popup&apos;s context. An attacker who can control product names or branch settings can plant a stored XSS payload that fires when any admin prints a report.`nFix`nSanitize all values before inserting into the HTML string. Use a simple HTML escape function, or use the DOM API to create elements rather than concatenating HTML strings.`nCode`n// ❌ CURRENT (raw interpolation):`n`n<h1>${title}</h1>`n`n<p>${metadata?.branch || &apos;Main&apos;} Branch</p>`n`n`n`n// ✅ FIX — escape helper:`n`nconst esc = (s) => String(s).replace(/&/g,&apos;&amp;&apos;).replace(/</g,&apos;<&apos;)`n`n                           .replace(/>/g,&apos;>&apos;).replace(/&quot;/g,&apos;&quot;&apos;);`n`n<h1>${esc(title)}</h1>`n`n<p>${esc(metadata?.branch || &apos;Main&apos;)} Branch</p>`n`n#10 — Missing HTTP Security Headers  [HIGH]`nAffected File(s)`nbackend/server.js`nProblem`nThe server does not use Helmet.js or manually set any security headers. This means browsers receive no guidance on: Content Security Policy (CSP), X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, or HSTS. Missing CSP in particular greatly increases the impact of any XSS vulnerabilities. Without X-Frame-Options, the app could be embedded in a malicious iframe.`nFix`nInstall and configure Helmet.js as early middleware.`nCode`nnpm install helmet`n`n`n`n// Add to server.js (before route definitions):`n`nimport helmet from &apos;helmet&apos;;`n`napp.use(helmet());`n`napp.use(helmet.contentSecurityPolicy({`n`n  directives: { defaultSrc: [&quot;&apos;self&apos;&quot;], scriptSrc: [&quot;&apos;self&apos;&quot;] }`n`n}));`n`n#11 — 50 MB JSON Body Limit  [HIGH]`nAffected File(s)`nbackend/server.js (~line 986)`nProblem`nbodyParser.json({ limit: &apos;50mb&apos; }) allows any client to send a 50 MB JSON payload. Code Quality Observations`n`nBeyond security, the following code quality issues were noted:`n`nDual schema files — Both schema.sql (root) and backend/schema.sql exist and have diverged slightly. The root schema is slightly more complete (includes alert settings columns). Maintain only one canonical schema file.`nrefreshData() fetches all data on every action — The App.tsx refreshData() re-fetches products, logs, users, sales, and settings on every minor action. For a small local store this is acceptable, but as data grows this will degrade performance. Consider selective invalidation.`nDashboardView.tsx hardcodes the low stock threshold — Products are flagged as low stock when stock <= 3, ignoring the lowStockThreshold setting from the database. This is a business logic inconsistency parallel to the hardcoded margin bug.`nprintRevisedReceipt() in SalesReport.tsx also uses document.write() with raw sale data — Same XSS risk as printService.ts; apply the same escaping fix.`nError messages reveal internal details — handleError() in production mode correctly omits stack traces, but err.message is still forwarded to clients. Prefer generic messages for unexpected errors.`nNo input sanitization on product text fields — Fields like name, specs, and detailedDescription are stored and retrieved as-is. While they are currently HTML-escaped by React on render, the print window does not benefit from this. Sanitize on write.`n`n7. Conclusion`n`nThe JewelAdmin Pro codebase has made significant security progress since the original February 2026 audit. The most dangerous vulnerabilities from that report — SQL injection, hardcoded credentials, MD5 hashing, and missing JWT auth — have been corrected.`n`nHowever, 8 issues remain at CRITICAL or HIGH severity that could allow an attacker on the local network to forge audit logs, crack the default administrator password, circumvent PIN protection, or execute cross-site scripting payloads through the print service.`n`nGiven this system handles financial transactions and employee accountability records for a jewelry store, these issues should be treated with urgency. The fixes required for Phases 1 and 2 collectively represent less than a day&apos;s work and should be completed before any wider network deployment or before allowing untrusted devices onto the same LAN.`n`nOverall Risk Level: HIGH — Do not deploy on internet-facing infrastructure until all CRITICAL items are resolved.`n
